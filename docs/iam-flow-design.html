IDP - IAM Flow & Table Design Guide

A structured guide to designing the Identity & Access Management foundation for the Internal Developer Portal

The Engineering Method: Flow → Tables → Schema

Most beginners start with the schema and try to fit flows into it. That produces bad designs. You start with flows and let the schema emerge.

Part 1: The Phase 1 Flow (IAM Foundation)

These are the concrete user actions your system must support. Read them, then ask: "What tables do I need to make this work?"

Flow 1.1

User Onboarding (Self-Registration)

  1. User opens the portal → sees login page
  2. User clicks "Sign Up"
  3. User submits: email, password, display name
  4. System creates User record (status: PENDING_VERIFICATION)
  5. System sends verification email with token
  6. User clicks link in email
  7. System marks email as verified (status: ACTIVE)
  8. User can now log in

What data is created?

  • User row (with hashed password, email_verified=false)
  • EmailVerificationToken row (or just a token field)
  • AuditLog entry: UserRegistered

What tables are touched?

  • users
  • email_verifications (or embedded in users)
  • audit_logs
Flow 1.2

Login + Session

  1. User submits email + password
  2. System verifies password (constant-time compare)
  3. System checks: email verified? account active? not locked?
  4. System creates Session row (with expiry)
  5. System sets HttpOnly cookie with session ID
  6. System creates AuditLog entry: UserLoggedIn
  7. User is redirected to dashboard

What tables are touched?

  • users (read)
  • sessions (insert)
  • audit_logs (insert)
  • login_attempts (optional, for rate limiting)
Flow 1.3

Authenticated Request (Every API Call)

  1. Browser sends request with session cookie
  2. Middleware reads cookie → looks up session
  3. Middleware checks session expiry
  4. Middleware loads user + their org memberships + roles
  5. Middleware populates request context
  6. Handler runs authorization check
  7. Handler performs the action
  8. AuditLog entry created for sensitive actions

What tables are touched?

  • sessions (read, possibly update last_seen)
  • users (read)
  • organization_members (read)
  • roles (read)
  • permissions (read)
  • audit_logs (insert for sensitive actions)
Flow 1.4

Organization Creation

  1. Authenticated user clicks "Create Organization"
  2. User submits: name, slug
  3. System creates Organization row
  4. System creates OrganizationMember row (user → org, role=OWNER)
  5. System creates AuditLog entry: OrganizationCreated
  6. System optionally creates default Project, Environments

What tables are touched?

  • organizations (insert)
  • organization_members (insert)
  • audit_logs (insert)
  • projects (insert, if auto-create)
  • environments (insert, if auto-create)
Flow 1.5

Inviting a Member

  1. Org Owner enters invitee's email
  2. System creates Invitation row (token, expiry, role)
  3. System sends invitation email
  4. Invitee clicks link → registers or logs in
  5. System creates OrganizationMember row
  6. Invitation marked as ACCEPTED

What tables are touched?

  • invitations (insert, update)
  • organization_members (insert)
  • audit_logs (insert)
Flow 1.6

Role-Based Access Check

  1. User attempts: "Delete project X in org Y"
  2. System checks: does user have membership in org Y?
  3. System checks: does user's role have permission "project:delete"?
  4. If yes → proceed
  5. If no → 403 Forbidden
  6. AuditLog entry: AuthorizationDenied (optional)

What tables are touched?

  • organization_members (read)
  • roles (read)
  • role_permissions (read)
  • permissions (read)
  • audit_logs (insert on denial, optional)

Part 2: The Future Flows (Phases 2-5)

Now think ahead: what tables do these flows need?

Flow 2.1 Phase 2

Configuration Management

  1. Project admin navigates to env "production"
  2. Admin adds config: key=API_URL, value=https://api.example.com
  3. System creates Configuration row (version=1)
  4. Admin updates value → new version=2, old version preserved
  5. Admin rolls back to version 1 → new version=3 with old value
  6. AuditLog entry for each change

Tables needed:

  • configurations
  • configuration_versions (or version column on configurations)
Flow 2.2 Phase 2

Secret Management

  1. Admin adds secret: key=DATABASE_PASSWORD, value=***
  2. System encrypts value, stores encrypted blob
  3. System creates Secret row (version=1)
  4. Admin reveals secret → system decrypts, returns value once
  5. AuditLog entry: SecretRevealed (CRITICAL - must log who, when, why)
  6. Admin rotates secret → new version=2
  7. Old versions preserved for retention period

Tables needed:

  • secrets (metadata + encrypted value reference)
  • secret_versions
  • secret_access_logs (or just audit_logs)
Flow 2.3 Phase 2

Deployment Tracking

  1. Developer triggers deployment to "staging" environment
  2. System creates Deployment row (status=PENDING)
  3. Worker picks up job, starts deployment
  4. System updates: status=RUNNING, started_at=now
  5. Deployment succeeds → status=SUCCEEDED, completed_at=now
  6. Or fails → status=FAILED, failure_reason="..."
  7. AuditLog entry for deployment action

Tables needed:

  • deployments
  • deployment_events (or just status history)
Flow 2.4 Phase 1+

Project + Environment Management

  1. Org Owner creates Project
  2. System auto-creates default environments: dev, staging, production
  3. Each environment has a protection level
  4. Each environment can have configs, secrets, deployments
  5. Production requires stricter permissions

Tables needed:

  • projects
  • environments
  • project_members (if you want project-scoped roles)
Flow 2.5 Phase 2

Background Jobs

  1. System needs to send invitation email
  2. System enqueues job: { type: 'send-invitation', payload: {...} }
  3. Worker picks up job, processes it
  4. Worker updates job status
  5. On failure, retry with backoff
  6. After N retries, move to dead-letter queue

Tables needed:

  • jobs (or use BullMQ's built-in storage in Redis)
  • job_history (or just logs)
Flow 2.6 Phase 2

Notifications

  1. System event occurs (e.g., deployment succeeds)
  2. System creates Notification row for relevant users
  3. Users see notifications in their inbox
  4. User marks notification as read

Tables needed:

  • notifications
  • notification_preferences (per-user settings)
Flow 2.7 Phase 1+

Audit Log Querying

  1. Admin opens audit log viewer
  2. Admin filters: org=acme, action=secret.revealed, date_range=last-week
  3. System queries audit_logs with filters
  4. Admin can export results

Tables needed:

  • audit_logs (with proper indexes for filtering)
Flow 2.8 Phase 3

API Catalog

  1. Project admin registers an API: name, version, base URL, owner
  2. System creates ApiCatalogEntry row
  3. Other users can search/filter the catalog
  4. Future: import from OpenAPI spec files

Tables needed:

  • api_catalog_entries
Flow 2.9 Phase 3

Infrastructure Registry

  1. Project admin registers a resource: type=database, name=users-db, env=production
  2. System creates InfrastructureResource row
  3. Admin can add dependencies: this resource depends on that one
  4. Future: automated discovery

Tables needed:

  • infrastructure_resources
  • infrastructure_dependencies (or use a graph table)
Flow 2.10 Phase 3

Logs & Metrics Viewer

  1. Developer opens logs viewer for environment "production"
  2. Developer filters by severity=ERROR, time_range=last-hour
  3. System displays structured logs (with sensitive values masked)
  4. Developer clicks correlation ID → see all related logs

Tables needed:

  • application_logs (or use external log store like CloudWatch)
  • log_retention_policies (optional)
Flow 2.11 Phase 2

Health Checks

  1. System periodically checks: database, queue, external dependencies
  2. Health check worker writes to health_check_results table
  3. UI displays "All systems operational" or specific degradation
  4. Historical health data available for trend analysis

Tables needed:

  • health_checks (definition: what to check)
  • health_check_results (history: when, status, duration)
Flow 2.12 Phase 3

Multi-Cloud Deployment Tracking

  1. Org wants to deploy some resources to AWS, others to GCP
  2. System tracks which provider each resource uses
  3. Cost comparison per provider
  4. Future: automated deployment to multiple clouds

Tables needed:

  • (probably extends infrastructure_resources with provider column)
  • cost_records

Part 5: The Key Design Decisions to Make Yourself

These are the decisions that will shape your schema. Think through each one.

Decision 1: How do you model user identity?

Option A: Single users table
id, email, password_hash, email_verified, status, created_at
Option B: Separate auth_identity table
users (profile) + identities (auth methods: password, OAuth, etc.)
More flexible, supports multiple auth methods per user. More complex, not needed for Phase 1.
For Phase 1, Option A is fine. Option B is a Phase 2+ improvement.

Decision 2: How do you model org membership?

Option A: organization_members table (user_id, org_id, role_id, status)
Simple many-to-many. One user can be in many orgs.
Option B: Separate membership_invitations + memberships
Invitations and memberships as separate concepts. Cleaner audit trail.
Option C: Membership with embedded role (no separate Role table)
organization_members has role as enum. Simplest, but less flexible.
For Phase 1, Option A is the standard pattern. Decide if role is an enum (simple) or a table (flexible).

Decision 3: How do you model permissions?

Option A: Hardcoded permission strings
'project:create', 'secret:reveal', etc.
Checked in code. Simple, but changes require code deploy.
Option B: Permissions in database (roles + role_permissions + permissions)
Database-driven, changes don't need code deploy. More flexible, but more complex. What your PRD implies (permission matrix).
Option C: Hybrid (default permissions seeded in DB, can be customized)
Most flexible. Overkill for Phase 1.
For Phase 1, Option B matches your PRD's permission matrix. Seed it with the 6 default roles from your PRD.

Decision 4: How do you model sessions?

Option A: Full session row in DB
session_id, user_id, ip, user_agent, expires_at, created_at
Server can revoke instantly. DB hit on every request (or cache in Redis).
Option B: Stateless JWT
Token contains user info, signed. No DB hit, but can't revoke instantly.
Not recommended for secrets-managing platform.
Option C: Session ID + Redis storage
session_id in cookie, session data in Redis. Fast, revocable, scales well. Best of both worlds.
For Phase 1, Option A (DB-backed) is simplest. Option C is the serverless-ready upgrade.

Decision 5: How do you model audit logs?

Option A: Single audit_logs table for everything
All events in one table, filter by entity_type. Simple, one place to query.
Option B: Separate tables per entity (project_audit, secret_audit, etc.)
Cleaner per-entity schemas. Harder to query "all events for user X."
Option C: Hybrid (event store pattern)
events table (immutable) + projections. Most powerful, most complex.
For Phase 1, Option A is standard. Your PRD says "filter by organization, project, actor, action, date range" — one table works.

Decision 6: How do you handle soft delete?

Option A: deleted_at timestamp on every table
Soft delete by setting timestamp. Filter WHERE deleted_at IS NULL everywhere. Simple, works.
Option B: Separate archive tables
Active data + archive data. Cleaner performance for huge tables. More complex.
Option C: Status enum (ACTIVE, ARCHIVED, DELETED)
More explicit than deleted_at. PRD uses this for Organization/Project states.
For Phase 1, Option C matches your PRD's state machines. Use a status field where appropriate.

Decision 7: How do you model environment protection levels?

Option A: Enum (LOW, MEDIUM, HIGH, CRITICAL)
Simple, fits PRD's "production requires stricter permissions." Stored as integer or string.
Option B: Separate protection_level table
More flexible (can add custom levels). Overkill for Phase 1.
Option A. Match the PRD.

Decision 8: How do you track who created/updated a record?

Option A: created_by_id, updated_by_id on every table
Foreign key to users. Standard audit fields.
Option B: Rely on audit_logs for this info
Less duplication. But harder to query "show me all projects I created."
For Phase 1, Option A on critical tables (orgs, projects, environments, configs, secrets).

Part 7: The Schema Design Exercises

Now you design. Here's a structured way to do it.

Exercise 1: List Your Entities

Write down every entity from Part 4 (the 11 tables for Phase 1). Don't worry about columns yet.

Exercise 2: List The Columns For Each

For each entity, ask: "What facts do I need to know about this?"

Example for User:

  • id (unique identifier)
  • email (login)
  • password_hash (authentication)
  • display_name (UI)
  • email_verified (state)
  • status (active/suspended/pending)
  • created_at, updated_at (audit)
  • last_login_at (optional, for security)
  • failed_login_count (for lockout)

Exercise 3: Identify Primary Keys

UUID vs auto-increment integer? Pros/cons of each?

Hint: UUIDs are better for distributed systems, but ints are simpler. Your call.

Exercise 4: Identify Foreign Keys

For each entity, what other entities does it reference?

  • OrganizationMember → User, Organization, Role
  • AuditLog → User (actor), Organization (optional), etc.

Exercise 5: Identify Indexes

What queries will be frequent?

  • Login: WHERE email = ?
  • Session lookup: WHERE id = ?
  • Audit query: WHERE organization_id = ? AND created_at > ?
  • Membership check: WHERE user_id = ? AND organization_id = ?

Exercise 6: Identify Constraints

What rules must the database enforce?

  • UNIQUE on email
  • NOT NULL on critical fields
  • CHECK constraints (e.g., role must be valid)
  • Foreign key constraints with ON DELETE behavior

Exercise 7: Identify Enums vs Tables

  • Role: enum or table? (You decide based on Decision 3)
  • Permission: enum or table? (You decide)
  • Status: enum or table? (Usually enum)
  • Environment type (dev/staging/prod): enum? (Yes)

Part 8: The Future-Proofing Checklist

Before you write your schema, check it against these:

Concern How Your Schema Should Handle It
Multi-tenancy Every business entity has organization_id (or is scoped via org)
Soft delete status or deleted_at on entities that can be archived
Audit trail created_at, updated_at, created_by, updated_by on critical tables
Tenant isolation All queries include organization_id in WHERE clause
Future: configs/secrets Configuration and Secret will be similar; design them consistently
Future: deployments Deployment references project + environment
Future: notifications Notification references user + optional org/project
Future: multi-cloud Consider a provider column early on infrastructure_resources
Migration path Schema is in Prisma, so you can evolve with prisma migrate

Part 9: The Recommended Order to Build Your Schema

  1. User + Session — the absolute core
  2. Organization + OrganizationMember — tenancy
  3. Role + Permission + RolePermission — RBAC
  4. AuditLog — cross-cutting
  5. Invitation — membership growth
  6. EmailVerification + PasswordReset — auth flows

Then Later Phases

  1. Project + Environment
  2. Configuration
  3. Secret
  4. Deployment
  5. Notification
  6. ApiCatalogEntry
  7. InfrastructureResource

The One Question I Want You to Ask Yourself

Before you write a single line of Prisma schema, answer this:

"If I had to query 'all secret reveals in the last 30 days across all orgs,' what tables and what indexes would make that fast?"

If you can answer that confidently, your schema is well-designed. If you can't, you need to think more.

This is the kind of question senior engineers ask. It forces you to think about:

  • What tables exist
  • How they relate
  • What indexes support the query
  • Whether your tenant model works
